# LUCID stack interaction

**Project:** LUCID  
**Benchmark version:** 1.1.0 (active profile)  
**Document type:** Adjacent governance — relationship to sibling stacks  
**Status:** Canonical

---

## 1. Purpose

LUCID is a **benchmark**, not a reimplementation of RediAI, DARIA, Foundry, CLARITY, or other sibling systems. This document fixes how LUCID **borrows posture** without **merging identity**.

---

## 2. Posture map (inform, not import)

| Source | What LUCID borrows | What LUCID does not do |
|--------|--------------------|-------------------------|
| **CLARITY** | Evaluation discipline: controlled stress, metrics over narrative, robustness thinking | Rebuild CLARITY’s evaluation stack inside LUCID |
| **DARIA** | Deterministic runs, artifact bundles, reproducibility | Become a DARIA runtime |
| **Foundry** | Modularity, small reversible changes, clear roles | Reimplement Foundry orchestration |
| **RediAI** | Truthful CI, evidence-backed change, governance habits | Claim RediAI certification for LUCID by default |

---

## 3. Identity rule

- LUCID **may** reuse **patterns** (CI layout, doc hierarchy, hashing discipline).
- LUCID **must not** silently merge identities: a benchmark repo is not a solver repo and not a platform product.

---

## 4. Public communication

- Competition-facing copy should emphasize **LUCID’s benchmark story** and **metacognitive calibration**.
- Stack lineage may appear in **technical** or **governance** appendices; it should not **overshadow** the scientific claim of the benchmark.

---

## 5. One-line summary

> LUCID stands next to the stack for discipline and determinism; it does not absorb the stack’s identity or reimplement it.
